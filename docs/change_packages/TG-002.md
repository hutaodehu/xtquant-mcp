## Goal

统一 Trade Gateway 的单账户优先主契约，让 schema、resource、session 与执行路径表达一致。

## Implemented

1. 收口 public schema 为单账户主流程：
   - `session.warm` 不再接收 `account_id`，只保留 `force`。
   - `session.status` / `session.close` 不再接收 `account_id`。
   - `fills.list` 不再接收 `account_id`。
   - `miniqmt.ensure_logged_in` 也改为使用服务端主账户配置，不再暴露 `account_id` 输入。
2. server 运行时同步拒绝显式账户选择，而不是只改 schema 文案：
   - 对 `miniqmt.ensure_logged_in`、`session.*`、`fills.list` 及相关主流程工具增加字段校验。
   - 旧式 `account_id` 输入现在会返回 `validation_error`，避免 schema 与执行路径漂移。
3. 执行路径统一到服务端主账户上下文：
   - `session.warm()` 默认从服务端配置或 `auto_account` 建立主账户上下文，不再依赖显式账户参数。
   - `fills.list` 改为直接使用当前 service context 的 `account_id`，不再从外部参数注入账户。
4. `GatewaySessionManager` 支持无显式账户预热：
   - 当 `trade.account_id` 为空且 `auto_account=true` 时，允许 `session.warm()` 通过 runtime auto-resolve 建立主账户 owner。
   - session summary / not-ready / mismatch / close 输出统一带上单账户主契约元数据。
5. 统一 resource/runtime 元数据：
   - 新增 `account_contract=single_account_primary`
   - 新增 `account_input_mode=service_context_only`
   - 新增 `account_scope`（如 `primary_session` / `primary_account`）
   - `trade://session/current`、`trade://account/current`、`trade://orders/today` 读取时都会补齐这些字段。
6. 统一 prompt 与 health 输出口径：
   - `/healthz` 增加账户契约元数据。
   - `trade-preflight` / `order-followup` 明确提示主流程使用 server-side primary account context，不做 per-call account switching。
7. 补充开发自测：
   - 更新 `tests/test_trade_gateway_server.py`
   - 新增 `tests/test_trade_gateway_session_manager.py`
   - 验证 schema、resource、validation reject 和 `auto_account` 预热行为。

## Known Risks

1. 这是收紧契约的兼容性变更：旧调用方若仍给 `session.*`、`fills.list` 或 `miniqmt.ensure_logged_in` 传 `account_id`，现在会收到 validation 错误。
2. `auto_account` 仍依赖底层 runtime resolution；若本机账号发现逻辑不稳定，`session.warm()` 仍可能落为环境失败而不是设计失败。
3. `miniqmt.ensure_logged_in` 现在也走服务端主账户配置，要求部署配置中的主账户信息保持正确；若部署长期依赖调用时覆盖账户，需要单独规划未来多账户扩展，不应回退当前主契约。
