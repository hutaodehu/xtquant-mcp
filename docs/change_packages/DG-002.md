## Goal

为 `xtdata://leases/active` 增加 experimental subscription lease 健康输出。

## Implemented

1. 将订阅输出重构为 experimental subscription lease 视图，而不是仅暴露原始 `subscription_id` / `seqs` 残留。
2. 为 lease 视图增加健康字段：`callback_loop_alive`、`connection_alive`、`last_event_at`、`last_event_age_seconds`、`needs_rebuild`、`rebuild_reason`、`lease_state`、`stop_reason`。
3. 在 lease 顶层与每个 lease 条目中关联当前 `resolved_runtime_endpoint` / `configured_endpoint`，使 agent 能把租约健康与当前运行时端点关联起来。
4. 为 `subscribe_start` / `subscribe_stop` 返回值补充 lease health 结构，并修复“回调先触发但订阅状态尚未注册”导致首个事件丢失的问题。
5. 新增 `xtdata://leases/active` 资源并加入默认 resources 列表；server 读取该资源时返回当前 lease health 视图，`xtdata.status` 与 subscribe 工具调用后也会刷新该资源缓存。
6. 将 subscription tool / prompt 文案明确标注为 experimental lease，避免误导 agent 把订阅能力当成稳定长期句柄。
7. 增加 tests 覆盖：active lease、stale lease（无事件到达）、leases resource 列表与读取。

## Not Implemented

1. 未证明自动重连或 lease 重建在真实环境已稳定成立；本卡只输出“是否需要重建”的判断，不声明重建链路可依赖。
2. 未将 subscription lease 升格为长期持久句柄；进程重启后遗留状态仍按 `process_restart` / `stopped` 处理。
3. 未对 `DG-001` 之外的数据状态模型做返工，也未进入任何 `VAL-*`。

## Known Risks

1. `needs_rebuild` 当前使用启发式阈值（grace/stale seconds）判断无事件或陈旧事件，尚未经过真实行情窗口长时验证。
2. `connection_alive` 当前基于当前 xtdata 端口探测，而不是逐 lease 独立 transport 诊断，因此只表示“当前后端连通性”，不等价于 broker 侧已确认恢复。
3. callback loop 活性仍以内存中 callback 注册状态为准；若底层库内部线程异常但回调对象仍存在，仍需独立 smoke 才能完全验证。
