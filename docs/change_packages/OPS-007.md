## Adapter Mapping Decisions

1. `task_truth_carriers`
   - 由 `docs/task_cards/<TaskID>.md` 与 `TaskCard` 中的 `Repo Spec Link` 目标共同承载。
   - `TaskCard` 负责执行边界和交接入口，spec 文档负责目标契约和语义边界。
2. `change_evidence_carriers`
   - 主体是 `docs/change_packages/<TaskID>.md`。
   - 被 ChangePack 明确回链的修改文件或运行态实现 artifact 可作为 change evidence 的底层载体。
   - ChangePack 中的自测只代表开发 truth，不等于独立测试通过。
3. `validation_evidence_carriers`
   - 主体是 `docs/evidence_packs/<TaskID>-<Role>-<YYYYMMDDHHMM>.md`。
   - 需要环境事实时，由 `docs/env_snapshots/<TaskID>-<YYYYMMDDHHMM>.md` 补充。
   - 原始日志、probe、state snapshot、resource readback 必须被 EvidencePack 显式回链后，才进入正式验证证据链。
4. `final_release_sources`
   - 主体是 `docs/reviews/<TaskID>-review-<YYYYMMDDHHMM>.md`。
   - 同时依赖 `docs/ACCEPTANCE_STANDARD.md` 中适用 gate，以及任务需要的外部 approval artifact。
   - `RunLedger` 的 `Accepted` / `Review Result` 只能镜像 release 结论，不能反向充当 source。
5. `env_snapshot_sources` 及 related fields
   - `env_snapshot_sources` 由 `docs/env_snapshots/<TaskID>-<YYYYMMDDHHMM>.md` 承载。
   - `runtime_health_sources`、`runtime_state_sources`、`resource_sources` 由 EvidencePack / EnvSnapshot 回链的具体 runtime artifact 承载。
   - `staleness_sources` 由 EvidencePack、EnvSnapshot、原始 artifact 的时间戳/TTL/recency 信息，以及验收标准或任务 spec 中的 freshness 要求共同承载。
6. ledger / controller boundary
   - `Board Export` 是 reconcile 输入，不是 truth carrier。
   - `Board Sync` 和 controller judgment 是 control-plane 行为，不是角色工件，也不新增 authority。
   - board 字段只索引正式工件和阶段状态，不替代底层 artifact。
7. generic example tightening
   - 通用的 TaskCard/ChangePack-style adapter 示例不再把 `board/review gate` 当作 `final_release_sources`。
   - 若 release authority 通过外部流程体现，adapter 应指向正式 approval artifact 或 release gate owner artifact，而不是指向 ledger 字段。

## Known Risks

1. 当前 adapter 映射仍是文档级 contract，而不是 machine-readable manifest；后续若要自动消费这些字段，仍需要单独实现卡把 prose contract 固化成结构化输出。
2. `runtime_health_sources`、`runtime_state_sources`、`resource_sources` 的具体 carrier 仍依赖 lane-specific spec 和 EvidencePack 写法；如果后续任务不按本次规则显式回链 artifact，仍会出现证据链漂移。
3. ReviewPack 之外的 external approval artifact 目前尚未统一命名或统一目录；高风险场景若不在 ReviewPack 中明确回链，仍可能把“release gate owner artifact”写成抽象流程名词而不是具体 artifact。
4. generic example 现已与当前仓库边界对齐，但其他 repo 若真的把 board 兼作 approval artifact 容器，后续 adapter 仍必须明确指向该 artifact 本体，否则读者仍可能偷换成状态字段。
