## Goal

为 MiniQMT UI 可见性探测补一个最小安全退化路径：

1. 当 OS 仍报告存在主窗口，但 `pywinauto` 枚举返回空时，`WindowsDesktopHarness.observe()` 不能把证据压缩成空集合。
2. 在不夸大“已登录”结论的前提下，把宿主层可见的顶层窗口事实暴露到 observation evidence。
3. 截图捕获要么成功输出路径，要么在 evidence 中明确报告已尝试及失败原因。

## Implemented

1. 在 `xtqmt_mcp/miniqmt_login/desktop_harness.py` 的 `WindowsDesktopHarness.observe()` 增加 host-visible top-level window fallback：
   - 当目标进程存在且 `pywinauto` 未枚举到任何窗口时，回退到 Win32 `EnumWindows` / `GetWindowTextW` / `GetClassNameW` / `GetWindowThreadProcessId` 采集顶层窗口事实。
   - fallback 只补证据与主窗口候选识别，不把“看到了主窗口”提升成“登录成功”结论。
2. 为 `WindowDescriptor` 增加 `class_name`，并把宿主层窗口事实写入 observation evidence：
   - `interactive_desktop`
   - `pywinauto_window_count`
   - `host_window_fallback_used`
   - `host_visible_windows`
   - `window_classifications`
3. 调整截图采集为可报告模式：
   - 新增 `capture_screenshot_details()` 返回 `path / attempted / error`
   - `observe()` 在 evidence 中输出 `screenshot_capture_attempted` 与 `screenshot_capture_error`
   - 保留原 `capture_screenshot()` 作为兼容包装，不扩大调用面
4. 新增定向单测 `tests/test_miniqmt_desktop_harness.py`：
   - 覆盖 `pywinauto` 空枚举时的 Win32 fallback 证据输出
   - 覆盖截图尝试失败时的 evidence 报告

## Known Risks

1. Win32 fallback 只能证明“宿主层看到了该进程的顶层窗口”，不能证明控件树可操作，也不能单独证明登录成功。
2. `EnumWindows` 返回的是宿主当前时刻可见的顶层窗口快照；若窗口瞬时切换或最小化，evidence 仍可能与后续操作窗口不同步。
3. 截图 evidence 现在会显式报告失败原因，但如果运行会话没有图形抓屏权限，仍只能得到失败记录而非图像文件。

## Needs Independent Test

1. 在真实 MiniQMT 主机上复现“`MainWindowHandle != 0` 但 `pywinauto` 空枚举”的场景，确认 fallback evidence 与宿主窗口标题一致。
2. 验证上层调用方不会把 `main_window_found=true` 误读成“已登录且可交易”。
3. 验证受限桌面或锁屏场景下的截图失败 evidence 能被测试/审查侧正确分类为环境问题而非设计问题。
