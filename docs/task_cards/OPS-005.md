## Goal

把当前 repo-local 的通用交付技能设计包，从“研究与草案”收口为可继续拆卡执行的正式 spec 任务，并明确它与本仓库 `TaskCard`、`ChangePack`、`EvidencePack`、`ReviewPack`、`EnvSnapshot` 的 adapter 边界。

## Scope In

1. 固化 `workflow-router -> acceptance-analysis -> mode-* -> evidence-gate` 的职责边界、调用顺序和禁止动作。
2. 固化 `route_decision`、`risk_profile`、`acceptance_contract`、`gate_result` 的最小 contract，以及它们如何映射回当前仓库工件链。
3. 明确哪些内容继续保持 `repo-local` / design-only，哪些内容应拆成后续实现卡、验证卡或评审卡。

## Scope Out

1. 不在本卡内实现插件化、跨仓库分发或 Codex plugin packaging。
2. 不修改当前 Trade/Data Gateway 业务实现，也不改写现行 `G0 -> G4` 验收梯度。
3. 不在本卡内宣称通用技能套件已经可以正式替代当前 `spec-task-harness` 或现有 repo controller 流程。

## Done Means

仓库内对这套通用技能的正式执行入口不再是“实现整个 spec”，而是至少可以稳定回答以下问题：

1. 这套技能的 contract surface 是什么。
2. 它与当前仓库工件链如何映射。
3. 后续应该拆成哪些独立执行卡，而不是继续混成一个大任务。

## Notes

若后续 scope 扩张到真实 `.agents/skills/*` 行为变更、tool routing 改写或 eval 基线引入，应继续拆 follow-up 卡，而不是把实现与设计收口混在本卡里。

当前 analyze-only 拆分结果建议顺序为：

1. 先做 [OPS-006](./OPS-006.md)：收口通用 contract 与 authority 边界。
2. 再做 [OPS-007](./OPS-007.md)：把通用 contract 映射到当前仓库 `TaskCard/ChangePack/EvidencePack/ReviewPack/EnvSnapshot` adapter。
