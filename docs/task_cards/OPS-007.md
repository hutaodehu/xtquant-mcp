## Goal

把通用交付 suite 的通用对象，稳定映射到当前仓库的 `TaskCard`、`ChangePack`、`EvidencePack`、`ReviewPack`、`EnvSnapshot`、controller sync 与 repo policy 上，避免后续实现或评审时把抽象层和仓库本地 truth carriers 混读。

## Scope In

1. 定义当前仓库的 `task_truth_carriers`、`change_evidence_carriers`、`validation_evidence_carriers`、`final_release_sources`、`env_snapshot_sources` 等 adapter 映射。
2. 明确通用 suite 与当前仓库 `controller-only` / `controller-with-delegation`、board ledger、artifact chain 之间的边界关系。
3. 显式标出当前仓库 adapter 仍然无法稳定映射的缺口，避免后续实现阶段把缺口假装成已解决。

## Scope Out

1. 不修改 `spec-task-harness` 脚本行为，不修改 board sync JSON contract。
2. 不改变当前仓库 `G0 -> G4` 验收梯度，也不改写 Trade/Data/Validation lane 业务卡。
3. 不在本卡内引入真实 eval runner、CI workflow 或 plugin packaging。

## Done Means

后续执行者可以不靠聊天记忆，直接从仓库文档回答：

1. 通用 suite 的每个对象在当前 repo 中由谁承载。
2. 哪些 release / review / env truth 仍必须回到当前仓库专用工件。
3. 哪些 adapter 缺口需要继续拆成实现卡，而不是在 mapping 文档里含混处理。

## Notes

本卡必须在 [OPS-006](./OPS-006.md) 收口核心 contract 之后再推进；否则 adapter 会被迫绑定不稳定字段语义。
