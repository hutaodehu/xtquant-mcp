## Findings

1. High: “未登录导致 broker/session 未就绪”这个旧 blocker 已被登录后复跑正式关闭，不应继续作为 `VAL-004` 的当前真相口径。
   - baseline 证据 [VAL-004-test-202603311806-live-warm-fastfail-rerun.md](../evidence_packs/VAL-004-test-202603311806-live-warm-fastfail-rerun.md) 中，`session.warm` 仍以 `xttrader connect=-1` 失败，后续公共只读链路统一卡在 `session_not_ready`，结论是 `fail_env`。
   - 登录后复跑证据 [VAL-004-test-202603311831-post-login-rerun.md](../evidence_packs/VAL-004-test-202603311831-post-login-rerun.md) 中，同一 host / 同一 bundle / 同一实例目录下，`session.warm.ok=true`、`session.status.ready=true`、`probe.connection.ok=true`、`account.show.ok=true`、`positions.list.ok=true`、`snapshot.l1.ok=true`，且 `trade://session/current.ready=true`、`diag://probe/latest.ok=true`、`diag://login/latest.status=already_logged_in`。
   - 这说明旧的 env/login blocker 已被 reversal 证据关闭。若继续沿用“broker/session 未就绪”叙事，会把已经消失的环境症状误报成当前 blocker。

2. High: 当前 first blocker 应明确转为公共 `orders.list` 路径的 `fail_design`，不得与环境问题混写。
   - 登录后复跑里，`session.warm` 的 warm health 已能以 `source=xttrader_shadow`、`read_scope=warm_health_only` 成功读到 `orders.list count=3`。
   - 同一轮 ordered public chain 中，公共 `orders.list` 却立即以 `error_code=server_env_not_ready`、`error_message='NoneType' object has no attribute 'query_open_orders'` 失败。
   - 该差异说明当前 first blocker 不是“host 还没登录”或“session 仍未 ready”，而是 public `orders.list` 实现/契约路径本身存在空对象调用缺陷。按 [ACCEPTANCE_STANDARD.md](../ACCEPTANCE_STANDARD.md) 的分类，这属于状态语义或实现路径错误，应归为 `fail_design`，不是 `fail_env`。

3. High: `VAL-003` 仍不得启动，也不得宣称任何 `G4 ready`。
   - [VAL-003.md](../task_cards/VAL-003.md) 明确是 `G4` 受控最小真单验证卡，前提是 `G0 -> G3` 已闭合。
   - 当前 `G3` ordered public chain 仍在 `orders.list` 处失败，因此 `G3` 未重新闭合，`G4` 前置条件不成立。
   - `probe.connection.write_permission_ready=true` 只能证明 broker/write precheck 没有立即失败，不能覆盖公共 `orders.list` 的现存 `fail_design`，更不能外推成 `order.place` 写路径已 ready。

## Required Fix

1. 将当前 first blocker 明确登记为 public `orders.list` 路径的 `fail_design`，错误文本必须保持为 `'NoneType' object has no attribute 'query_open_orders'`，不得回退成“未登录/会话未就绪”的旧口径。
2. 后续修复应进入新的 bounded dev/test/review 闭环，聚焦 public `orders.list` 与 warm-health `orders.list` 之间的实现分叉；在新工件形成前，不得把该问题归咎为环境回退。
